# Промпт для Архитектора: Inline-кнопка "Завершить раунд" (AG-017)

**Роль:** Архитектор системы  
**Задача:** Проверить UX-решение на предмет влияния на архитектуру и состояния  
**Модель:** `modelstudio/qwen3-max-2026-01-23`

---

## Контекст

Есть Telegram-бот Agon Arena для тренировки управленческих поединков:
- 2 раунда с обменом ролями
- Пользователь vs AI-соперник
- После каждого раунда — разбор от 3 судей
- Общение текстом или голосовыми

**Текущая проблема:**
Кнопка "🏁 Завершить раунд" в нижнем меню скрывается при переписке. Не видна в нужный момент.

---

## Предложенное UX-решение

**Изменение:** Перенести кнопку из нижнего меню (ReplyKeyboardMarkup) в inline-кнопки (InlineKeyboardMarkup).

**Где показывать:**
1. Стартовое сообщение дуэли (после opening line)
2. Каждое сообщение с ответом AI-соперника
3. Сообщение перехода ко второму раунду

**CallbackData структура:**
```
duel:end:{duel_id}:{round_no}
```

**Тексты кнопок:**
- Раунд 1: "🏁 Завершить раунд"
- Раунд 2: "🏁 Завершить поединок"

**State-проверки в обработчике (6 проверок):**
1. Есть ли активная дуэль у пользователя
2. Совпадает ли duel_id
3. Совпадает ли текущий раунд
4. Не в состоянии ли дуэль уже finished/transitioning
5. Не идёт ли voice/STT/AI-processing
6. Выполнить transition

**Edge cases (5 сценариев):**
- Двойное нажатие → "Раунд уже завершён"
- Старая кнопка → "Эта кнопка уже неактуальна"
- Нет раунда → "Сейчас нет активного раунда"
- После таймаута → "Время уже истекло. Подвожу итог."
- Во время обработки голосового → "Сначала дождитесь обработки"

## Вопросы для архитектурного анализа

### 1. State Machine

**Текущие состояния дуэли:**
```
idle → round_1_active → round_1_processing → round_1_transition
     → round_2_active → round_2_processing → round_2_transition → finished
```

**Вопросы:**
- Достаточно ли текущих состояний для обработки callback?
- Нужно ли добавлять состояния типа `round_1_ending` или `pending_callback`?
- Как обрабатывать гонки (race conditions) при быстрых нажатиях?

### 2. Concurrency

**Сценарии:**
- Пользователь нажимает кнопку во время обработки голосового
- Пользователь нажимает кнопку дважды быстро (double-click)
- Пользователь нажимает старую кнопку из предыдущего раунда

**Вопросы:**
- Нужна ли блокировка на уровне duel_id?
- Как обрабатывать concurrent requests к одной дуэли?
- Достаточно ли текущей логики в `_process_end_round()` с `ACTION_IN_PROGRESS_USERS`?

### 3. Callback Flow

**Текущий обработчик:**
```python
@router.callback_query(F.data == "end_round")
async def end_round_or_finish_duel(callback: CallbackQuery) -> None:
    await _process_end_round(callback.message)
```

**Предлагаемое изменение:**
```python
@router.callback_query(F.data.startswith("duel:end:"))
async def handle_duel_end_callback(callback: CallbackQuery) -> None:
    # Parse: duel:end:{duel_id}:{round_no}
    # Validate state
    # Process transition
```

**Вопросы:**
- Не сломает ли это существующие callback handlers?
- Нужна ли миграция старых callback_data?
- Как обрабатывать старые кнопки, которые остались в истории чата?

### 4. Database / Persistence

**Текущая схема:**
- `Duel` таблица: status, current_round_number, final_verdict
- `DuelRound` таблица: status, started_at, finished_at
- `DuelMessage` таблица: история сообщений раунда

**Вопросы:**
- Нужно ли логировать callback события (для аудита/отладки)?
- Нужно ли хранить message_id кнопок для cleanup?
- Достаточно ли текущих полей для state validation?

### 5. Edge Cases & Error Handling

**Сценарии для проверки:**
- Кнопка нажата после перезапуска бота
- Кнопка нажата в другом чате (если бот в группе)
- Кнопка нажата во время миграции/деплоя
- Кнопка нажата при потере соединения с БД

**Вопросы:**
- Какие сценарии требуют дополнительной обработки?
- Нужны ли retry mechanisms?
- Как логировать ошибки для последующего анализа?

### 6. Влияние на другие компоненты

**Компоненты системы:**
- `duel_service.py` — логика дуэлей
- `round_timer_service.py` — таймеры раундов
- `judge_service.py` — разбор судей
- `transcription_service.py` — распознавание голоса

**Вопросы:**
- Влияет ли изменение на таймеры (нужно ли отменять таймер при нажатии кнопки)?
- Влияет ли на transcription (блокировать ли во время STT)?
- Влияет ли на judge flow (когда запускать судей)?
